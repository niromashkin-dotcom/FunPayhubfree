## 1.1. Цепочка запуска целиком (точка входа → готовая система)

### A. Хаб (`funpayhub_main.py` — Flask, «web service» на Render)
1. Поднимает `logs/app.log` (tee в stdout+файл) — `funpayhub_main.py:1-31`.
2. Чистит прокси из `os.environ`, настраивает WebView2 user-data — `:43-60`.
3. Если frozen (exe) — `os.chdir` в `_internal`, иначе в корень проекта и
   добавляет `ROOT` и `ROOT/web` в `sys.path` — `:63-80`.
4. Импортирует blueprint'ы: `plugin_management_api`, `alerts_api`, `logs_api`,
   `seller_api`, `userdata_api`, `funpay_proxy`, `assistant_api` — `:82-89`.
5. `init_plugin_system(plugins_dir="plugins", hub_url=...)` (`hub_bootstrap.py`)
   → создаёт `RuntimeLog`, `EventBus`, `PluginManager`, загружает плагины,
   `RuntimeController`, `NotificationManager`, `AutoReplyEngine`,
   `OrderPaymentTracker`, `OrderFlowManager` (Этап C), `EmergencyManager`
   (Этап J), `ReportEngine` (Этап D), `BackupManager` (Этап K),
   `SupplierWorkerPool` (Этап E), `AIEngineerAgent` (Этап G) — `:104-118`.
6. Регистрирует blueprint'ы — `:119-125`.
7. Регистрирует `/health` (текст `ok`) и `/api/version` — `:130-136`.
8. **API AUTH GUARD**: `_require_api_token` — `before_request` требует заголовок
   `X-API-Token` (из `FUNPAYHUB_API_TOKEN`) для всех `/api/*`. В headless без
   токена отдаёт 401 — `:138-174`.
9. Определяет `HOST`/`PORT`: headless → `0.0.0.0`, иначе `127.0.0.1`;
   `PORT` из `$PORT` — `:177-202`.
10. `signal.signal(SIGTERM/SIGINT, _handle_sigterm)` + `_shutdown_event`
    (graceful shutdown) — `:204-212`.
11. `main()`: Flask в daemon-потоке; прокси-проба `/static/dashboard.html`;
    в headless — удержание процесса через `_shutdown_event.wait(1)`;
    иначе pywebview/webbrowser — `:214-275`.

### B. EventBus (`eventbus.py`)
Синхронная шина `EventBus`: `subscribe/unsubscribe/emit`. `emit` копирует
список слушателей и вызывает каждый в `try/except` (ошибка слушателя НЕ
прерывает остальных, но логируется в `print`). **Нет** приоритета на самом
деле — аргумент `priority` принимается, но НЕ используется (порядок = порядок
подписки). `EventBus` создаётся в `hub_bootstrap`, затем инжектится в
`seller_service_singleton.event_bus` — `hub_bootstrap.py:194-212`.

### C. PluginManager (`plugins/plugin_manager.py`)
Конструируется с `HubStateAPI` (прокси к собственному Flask на `hub_url`,
TTL-кэш 5с) и `event_bus` — `hub_bootstrap.py:216-225`. `load_plugins`
читает `plugins/` — `:229-238`. Плагины (по `plugins/`): `autosmm_plugin`,
`autodonate_plugin`, `stars_plugin`, `autobump_plugin`, `telegram_notifier_plugin`,
+ вспомогательные (`config_manager`, `dependency_manager`, `logger_plugin`, …).

### D. Scheduler / фоновые потоки (`hub_bootstrap.py`)
Запускаются daemon-потоками (нет stop-сигнала — WARN из `PRODUCTION_READINESS`):
- **BackgroundCollector** (`_start_background_worker`) — 60с, вызывает
  `seller_service.collect_account_notifications()` — `:138-162`, `:388-395`.
- **MarketAutoUpdate** — старт + каждые 3ч, дёргает `/api/market/heatmap`,
  `/niches`, `/competitors`, `/ratings` — `:397-456`.
- **HealthCheck** — 60с, проверяет доступность поставщиков
  (gorgonaboosts/holdboost/kosell/shopclaude) и при смене статуса уведомляет
  админа + активирует/деактивирует лоты поставщика в `autodonate_plugin` —
  `:502-625`.
- **AutoBackup** — 6ч, копирует `configs/plugins/*.json` — `:460-495`.
- **OrderFlow** worker — 60с таймаут-проверка — `order_flow.py:553-562`.
- **ReportEngine** scheduler — 30с тик, в 6:00 и 21:00 — `report_engine.py:42-59`.
- **EmergencyManager** — только хранит состояние (без своего цикла).

> ⚠️ В репозитории **НЕТ** класса `BackgroundMonitor` и **НЕТ** рабочего
> `CacheManager` (`runtime/cache/cache_manager.py` и `__init__.py` —
> **пустые файлы 0 байт**). «Background Monitor» из вопроса 1.5 = совокупность
> перечисленных выше потоков. Кэширование реально живёт внутри
> `seller_service._get_account` (TTL) и `HubStateAPI` (TTL 5с), но в слое
> Telegram-бота его нет вообще (каждое нажатие → прямой HTTP к хабу).

### E. Telegram Bot (`run_bot.py` + пакет `bot/`)
1. `run_bot.py`: aiogram `Bot`+`Dispatcher`; `AuthMiddleware` (наружный);
   подключает роутеры `start`, `ai_agent`, `notifications`, `callbacks`;
   поднимает health HTTP-сервер на `$PORT` (Render требует открытый порт, иначе
   рестарты + 409 Conflict) — `run_bot.py:36-58`.
2. `ai_agent_service.configure(...)` — передаёт `LLM_API_KEY/URL/MODEL`
   (primary/secondary) + `GITHUB_TOKEN` + `RENDER_API_KEY/SERVICE_ID` —
   `:88-110`.
3. `ai_agent_service.start(bot)` — запускает `_scanner_loop` (скан `logs/app.log`
   каждые `_scan_interval`=300с) — `:111`.
4. `bot.delete_webhook` + `set_my_commands` + `dp.start_polling` — `:122-138`.

### F. Web API (`web/*.py` blueprint'ы)
Зарегистрированы в `funpayhub_main.py`. Все `/api/*` защищены `X-API-Token`
(см. 1.4). Это «мозг», к которому бот ходит через `bot/api_client.py`
(`APIClient`, таймаут `FUNPAYHUB_API_TIMEOUT`=8с).

**Ключевые расхождения с «ожидаемой» схемой (прямо по коду):**
- EmergencyManager и OrderFlowManager снимают лоты через HTTP к
  `/api/seller/lots/deactivate`, которого **нет** → авто-снятие при PAUSED и при
  нехватке баланса поставщика **фактически не работает** (исключение
  проглатывается). Рабочий механизм снятия — `POST /api/dev/lots/deactivate_all`
  (используется кнопкой «Снять все лоты» и `telegram_notifier_plugin`), но он
  **другой** и не принимает supplier. → Два разных механизма (п. 2.3).
- `OrderFlowManager._check_supplier_balance` ходит в
  `/api/seller/balance/suppliers` (нет) → исключение → `return True` → проверка
  баланса поставщика **никогда не срабатывает** → ветка «баланс 0 → CAPS-алерт +
  снятие лотов» (Этап C, шаг 2) недостижима из order_flow.
- `CacheManager` отсутствует (файл 0 байт). Слой бота не кэширует ответы.

## 1.6. Полный аудит AI-агента

В системе **два** «AI-агента» — это важно различать:

### (а) `bot.services.ai_agent_service` (управляется из Telegram, `run_bot.py`)
- **Модель:** OpenAI-совместимый API через `LLM_API_URL`/`LLM_API_KEY`/
  `LLM_MODEL` (env). Две конфигурации: `primary` и `secondary`
  (`run_bot.py:88-110`). Default модели: `mistral-small-latest` и
  `freellm-...` (длинный хардкод модели во вторичном — `run_bot.py:103`).
- **Fallback на другую модель:** ✅ ЕСТЬ. `_call_llm_with_fallback` перебирает
  список провайдеров (`ai_agent_service.py:131-141`). Реально работает, если
  заданы оба `LLM_API_KEY`/`LLM_API_URL`. ⚠️ Если вторичный ключ пустой —
  `ValueError("Empty API key")` и он пропускается; остаётся только primary.
- **Что читает:** сканирует `logs/app.log` (по `self._last_position`) на
  ключевые слова ERROR/Traceback/Failed (`_scan_logs`); плюс
  `scan_project_files` (TODO/FIXME/BUG по `*.py`, кроме секретных) и
  `analyze_project(file)` (конкретный файл). Секретные файлы блокируются
  `_is_sensitive_path` (`.env`, `*.key`, `*credential*`, `*secret*`,
  `*golden_key*`) — ✅ хорошо.
- **Что умеет:** анализировать ошибку → предложить патч (только JSON-совет,
  НЕ править код локально) → уведомить админа с кнопками «Да/Нет».
- **Применяет патч сам?** ❌ НЕТ. Только ПОСЛЕ подтверждения админом в Telegram
  (`confirm_apply_patch_*`). Даже тогда `apply_patch` НЕ меняет локальные файлы:
  он лишь (1) делает `GET`/`PUT` в GitHub (добавляет **комментарий** с текстом
  патча, не дифф), (2) триггерит Render-deploy (`_render_deploy`). Локальный
  код не патчится.
- **Rollback:** ❌ НЕТ. Нет механизма отката, т.к. локальные файлы не меняются;
  «патч» = комментарий в GitHub + деплой. Откат = ручной `git revert` (вне кода).
- **Кнопки в Telegram:** `ai_agent` (статус), `ai_scan_files`, `apply_patch_*`,
  `reject_patch_*`, `confirm_*`/`cancel_*`. Все обрабатываются в
  `bot/handlers/ai_agent.py`.

### (б) `runtime.ai_engineer_agent` (Этап G, в хабе)
- Запускается в `hub_bootstrap` только если `GOOGLE_API_KEY` задан. Сканирует
  ошибки логов → LLM → патч. По коду (`hub_bootstrap:374-386`) — тот же класс
  идей, источник ключа `GOOGLE_API_KEY`. Отдельного детального чтения не
  проводилось (вне основного флоу бота); отмечается как дублирующий (см.
  `PROJECT_FINAL_AUDIT` OPEN: «Дублирующие модули»).

### Честный вердикт по AI (1.6):
- ✅ Есть fallback-модель (primary/secondary) — реально перебирается.
- ✅ Не применяет патч без подтверждения (соответствует правилам безопасности).
- ❌ Не умеет реально патчить локальный код (только GitHub-комментарий +
  деплой).
- ❌ Нет rollback.
- ❌ Нет привязки патча к конкретному месту в коде (LLM возвращает произвольный
  текст, сохраняемый как комментарий).
