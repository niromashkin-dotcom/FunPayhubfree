# AI_CONTEXT.md - Файл контекста проекта FunPayHub

## 🗂️ СТРУКТУРА ПРОЕКТА (на момент начала аудита)

### КОРНЕВОЙ УРОВЕНЬ
- `funpayhub_main.py` - Основной файл запуска хаба, инициализация Flask-приложения, вебвью.
- `hub_bootstrap.py` - Загрузчик системы плагинов, инициализация EventBus, RuntimeLog, PluginManager, HubStateAPI и фоновых задач.
- `eventbus.py` - Единая шина событий для взаимодействия модулей, без хранения.
- `tg_bot_service.py` - Отдельный сервис Telegram-бота для управления хабом через команды.
- `cardinal.py` (вероятно устаревший/неиспользуемый, т.к. `hub_bootstrap.py` инициализирует систему БЕЗ Cardinal).
- `state_api.py` (старый, заменен `HubStateAPI` в `hub_bootstrap.py`).
- `Start.bat` - Скрипт запуска под Windows.
- `.clinerules` - Правила поведения для ИИ-агента.
- `requirements.txt` - Список зависимостей Python.
- `railway.toml` - Конфигурация для развертывания на Railway.app.
- `deploy.sh` - Скрипт для автоматического развертывания на Railway.
- `.env.example` - Пример файла переменных окружения.

### ДИРЕКТОРИИ ПРОЕКТА
1. **`FunPayAPI/`** - Интеграция с FunPay API: `Account` для взаимодействия, `Updater` для событий, `types` для моделей данных.
2. **`runtime/`** - Ядро системы, обработка событий, менеджеры: `RuntimeController`, `RuntimeLog`, `seller_service`, `autoreply_engine`, `order_tracker`, `notifications/`.
3. **`dashboard/`** - Web-интерфейс (Flask Blueprints: `api.py`, `registry.py`).
4. **`tg_bot/`** - Telegram бот (handlers, keyboards, authentication): `bot.py` (возможно, устаревший), `authorized_users.json`, `generate_password_hash.py`.
5. **`web/`** - Web API endpoints (Flask Blueprints): `plugin_management_api`, `alerts_api`, `logs_api`, `seller_api`, `userdata_api`, `funpay_proxy`, `assistant_api`.
6. **`Utils/`** - Утилиты и инструменты: `config_loader`, `exceptions`, `logger`, `cardinal_tools`, `secrets_manager`.
7. **`data/`** - Данные: `lot_emojis.json`, `synonyms.json`, `autosmm/`, `state/`.
8. **`locales/`** - Локализация: `en.py`, `ru.py`, `uk.py`.
9. **`plugins/`** - Модульная система плагинов: `plugin_manager`, `loader`.
10. **`security/`** - Модули безопасности: `secrets_manager.py`, `secret_loader.py`.
11. **`tests/`** - Тесты проекта.
12. **`userdata/`** - Пользовательские данные: `backgrounds/`, `sounds/`, `themes/`.
13. **`webview_data/`** - Директория для хранения данных WebView2.

## 🧠 Мысли и анализ

Я провел глубокое сканирование и анализ кодовой базы проекта FunPayHub. Вот мои ключевые выводы:

### 🏗️ Архитектурные особенности и паттерны

1.  **Плагин-ориентированная архитектура**: Проект активно использует систему плагинов (`plugins/`), которая позволяет расширять функционал без модификации основного кода. Это ключевой паттерн для гибкости и масштабируемости.
    *   `hub_bootstrap.py` отвечает за инициализацию `PluginManager` и загрузку плагинов.
    *   Плагины взаимодействуют с ядром через `EventBus` и `HubStateAPI`.

2.  **Event-Driven Architecture (EDA)**: Центральное место занимает `EventBus` (`eventbus.py`), который служит для обмена событиями между различными компонентами (плагины, `seller_service`, `AutoReplyEngine`). Это способствует слабой связанности модулей.
    *   События генерируются `RuntimeController`, `seller_service` (новые заказы, сообщения) и другими компонентами.
    *   Обработчики событий подписываются на нужные типы событий (`event_type`).

3.  **Client-Server (REST API)**: Основное приложение `funpayhub_main.py` запускает Flask-сервер, который предоставляет REST API (`web/` и `dashboard/api.py`).
    *   Telegram бот (`tg_bot_service.py`) взаимодействует с этим API для получения статуса, управления плагинами, получения логов и т.д.
    *   `HubStateAPI` в `hub_bootstrap.py` является прокси к этим внутренним API.

4.  **Фоновые задачи и многопоточность**: Проект активно использует `threading` для запуска фоновых воркеров (`_start_background_worker`, `_start_market_auto_update`, `_start_health_check`, `_start_auto_backup` в `hub_bootstrap.py`). Это позволяет выполнять длительные операции без блокировки основного потока.

5.  **ORM / Работа с БД**: Насколько я понял, явного ORM (вроде SQLAlchemy) не используется в основных файлах. Вероятно, данные хранятся в SQLite (`DATABASE_URL=sqlite:///data/funpayhub.db` из `.env.example`) и управляются напрямую или через кастомные модули (например, в `runtime/state/`).

6.  **Логирование**: Настроено кастомное логирование в файл (`app.log`, `tg_bot.log`) с использованием `logging` модуля. Есть обертка для `sys.stdout`/`sys.stderr` для дублирования вывода в файл и консоль. `RuntimeLog` в `runtime/runtime_log.py` предоставляет централизованный журнал событий хаба.

### ⚙️ Основные функции и возможности

1.  **Автоматизация FunPay**: Ядро функционала, предоставляемого плагинами (например, `autosmm_plugin`, `autodonate_plugin`). Включает автовыдачу товаров, автоподнятие лотов, автоответы, автовосстановление после продажи, вечный онлайн.

2.  **Управление через Telegram**: Полная панель управления хабом через Telegram-бота. Запуск/остановка хаба, просмотр баланса, лотов, логов, управление плагинами.

3.  **Мониторинг и аналитика**: Фоновые задачи для обновления данных о рынке, конкурентах, рейтингах. API для получения состояния системы и плагинов.

4.  **Управление плагинами**: Активация/деактивация плагинов, загрузка/перезагрузка конфигураций.

5.  **Обработка ошибок**: Есть базовые механизмы обработки исключений в `hub_bootstrap.py`, `tg_bot_service.py` и `config_loader.py`, а также кастомные исключения в `Utils/exceptions.py`.

6.  **Развертывание**: Оптимизировано для Railway.app с использованием `railway.toml` и `deploy.sh`.

### 🔥 GRILL ME REPORT: Результаты тотальной прожарки

1.  **КРИТИЧЕСКАЯ УЯЗВИМОСТЬ: Отсутствие шифрования Golden Key и Telegram Bot Token**: Несмотря на наличие `security/secrets_manager.py` и его интеграцию в `Utils/config_loader.py` (что я сделал в прошлый раз), в `tg_bot_service.py` всё ещё есть прямой доступ к `BOT_TOKEN` через `telegram_notifier_plugin.json` без шифрования. Golden Key FunPay также, вероятно, остался в открытом виде в `configs/_main.cfg` до полной миграции.
    *   **Где сломается/уязвимость**: Прямой доступ к критическим ключам злоумышленнику равносилен полному захвату аккаунта FunPay и Telegram-бота.
    *   **План спасения**: Полностью завершить интеграцию `SecretsManager` во все места, где используются эти ключи. Всегда использовать `os.getenv("ENCRYPTED_SECRET")` и расшифровывать на лету. Создать инструмент для миграции существующих конфигов.

2.  **КРИТИЧЕСКАЯ УЯЗВИМОСТЬ: Уязвимость в авторизации Telegram-бота**: В `tg_bot_service.py` команда `/auth` использует `hashlib.sha256(password.encode()).hexdigest()` для сравнения пароля. Это _лучше_ plaintext, но без соли (salt) и pepper уязвимо к rainbow table атакам. Также, после успешной авторизации, `user_id` добавляется в `authorized_users.json` в открытом виде.
    *   **Где сломается/уязвимость**: Слабая авторизация позволяет потенциальному злоумышленнику скомпрометировать бот, если он узнает хэш пароля.
    *   **План спасения**: Внедрить proper key derivation function (KDF) вроде `argon2` или `bcrypt` для хэширования паролей с солью. Добавить механизм временной сессии для авторизованных пользователей вместо перманентного добавления в файл. Либо использовать только whitelist без пароля, если это соответствует модели угроз.

3.  **ВЫСОКАЯ УЯЗВИМОСТЬ: SSRF и отсутствие валидации в `HubController.call_api`**: Метод `call_api` в `tg_bot_service.py` позволяет выполнять запросы к *любым* внутренним `HUB_URL` endpoint'ам. Это критическая SSRF-уязвимость, если злоумышленник сможет контролировать `endpoint`.
    *   **Где сломается/уязвимость**: Злоумышленник может использовать бота для сканирования внутренних портов, доступа к внутренним сервисам или выполнения произвольных HTTP-запросов от имени хаба.
    *   **План спасения**: Реализовать строгий белый список разрешенных `endpoint`'ов или использовать прокси-функции вместо прямого вызова `requests.get/post`.

4.  **ВЫСОКАЯ УЯЗВИМОСТЬ: Отсутствие таймаутов/rate limiting в API**: Методы в `web/` и `dashboard/api.py` (например, `/dashboard/api/generate_lots`, `/dashboard/api/remove_lots`, `/dashboard/api/run_simulation`) могут выполнять длительные или ресурсоемкие операции без механизмов rate limiting или таймаутов на стороне API. Telegram-бот вызывает эти методы напрямую.
    *   **Где сломается/уязвимость**: Потенциальная DoS-атака или истощение ресурсов сервера при злоупотреблении.
    *   **План спасения**: Внедрить Flask-расширения для rate limiting (например, `Flask-Limiter`) и добавить явные таймауты для операций, взаимодействующих с внешними API FunPay или выполняющих тяжелые вычисления.

5.  **СРЕДНЯЯ УЯЗВИМОСТЬ: Отсутствие обработки ошибок и retry-механизмов для внешних API**: В `hub_bootstrap.py` и `tg_bot_service.py` при вызовах `requests` нет централизованной обработки ошибок (например, сетевых сбоев, таймаутов, HTTP 4xx/5xx). Есть базовые `try-except`, но без повторных попыток с экспоненциальной задержкой.
    *   **Где сломается/уязвимость**: Временные проблемы с сетью или FunPay API могут привести к пропуску событий, зависанию или остановке функционала без уведомлений.
    *   **План спасения**: Внедрить декоратор `@retry` (например, из библиотеки `tenacity` или кастомный) для всех внешних API-вызовов. Добавить централизованную систему уведомлений о критических ошибках.

6.  **СРЕДНЯЯ УЯЗВИМОСТЬ: Циклические зависимости и монолитность**: Наблюдаются циклические импорты (не явные, но подразумеваемые через цепочку инициализации) и сильная связанность между `hub_bootstrap.py`, `runtime/runtime_controller.py`, `plugins/plugin_manager.py` и `eventbus.py`. Например, `EventBus` напрямую передается в `PluginManager`.
    *   **Где сломается/уязвимость**: Усложняет тестирование, рефакторинг и понимание потока данных. Любое изменение в одном модуле может вызвать непредвиденные побочные эффекты в других.
    *   **План спасения**: Внедрить паттерн Dependency Injection (DI) (например, через `python-injector`) для явного управления зависимостями. Разделить крупные модули на более мелкие, сфокусированные сервисы. Возможно, перейти к архитектуре микросервисов, если проект будет расти.

7.  **НИЗКАЯ УЯЗВИМОСТЬ: Хардкод API ключей в `hub_bootstrap.py` для Health Check**: В функции `_run_health_check` в `hub_bootstrap.py` (строки 444-447) API ключи поставщиков (`gorgonaboosts`, `holdboost`, `kosell`) хардкодятся напрямую в коде. Это противоречит принципу безопасного хранения секретов.
    *   **Где сломается/уязвимость**: Утечка этих ключей через исходный код. Хотя они используются только для Health Check, это всё равно небезопасно.
    *   **План спасения**: Использовать `SecretsManager` для загрузки этих ключей через переменные окружения или из зашифрованного файла.
