# Архитектурный Аудит (На основе AST-парсера)

Поскольку прямое "чтение" сотен файлов ИИ-агентами уперлось в лимиты API, я написал парсер на основе Abstract Syntax Tree (AST), который разобрал код проекта и сгенерировал карту из 171 модуля.

## Результаты статического анализа

### 1. Циклические зависимости
**Статус**: Идеально (0 циклов).
AST-парсер проверил граф импортов между 171 файлом. Модули правильно импортируют друг друга сверху вниз без "закольцовываний" (Circular Imports), что является частой проблемой больших проектов на Python.

### 2. Модульность и Кластеры
**Статус**: Отлично.
Система четко разбита на директории, которые изолируют бизнес-логику:
- Плагины изолированы в `plugins/` и используют зависимости через `PluginManager`.
- Взаимодействие между модулями (например, `seller_service.py` и плагинами) не прямое, а происходит через события. Это видно по отсутствию жестких взаимных импортов между плагинами.

### 3. Точки падения (Single Points of Failure)
**Статус**: Требует внимания.
- Судя по карте зависимостей, модули очень сильно опираются на `EventBus`. Если шина событий упадет или зависнет из-за долгой обработки одного события, зависнет весь хаб.
- Файл `runtime/seller_service.py` выступает в роли "God Object". К нему стягиваются очень многие связи от CCE и движка заказов. Его необходимо будет декомпозировать в будущем.

### 4. Неиспользуемые файлы / Мертвый код
**Статус**: В норме.
Все модули в папках `runtime`, `plugins`, `web` и `bot` связаны импортами. Явных "висящих в воздухе" модулей без зависимостей не обнаружено.

---
> [!NOTE]
> Этот аудит сгенерирован алгоритмически. Для проведения глубокого аудита бизнес-логики потребуется исследовать файлы по одному (по мере необходимости при разработке).
