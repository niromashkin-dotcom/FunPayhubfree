# Архитектура Проекта (Control Center 2.0)

Проект построен как **Controlled Refactoring Modular Monolith**. Это означает, что старый монолит (`seller_service.py`) стал "Фасадом" для новых модулей, чтобы не ломать обратную совместимость с плагинами.

## Общая Схема
```text
                     🛡 REAL MODE SAFETY
              (режим работы с реальными деньгами)

                           │

                    🧠 EventBus
                Нервная система проекта

                           │

 ┌──────────┬──────────┬──────────┬──────────┬──────────┐
 │          │          │          │          │          │
Orders     CCE       Lots      Ledger    Delivery     Plugins
 │          │          │          │          │          │
(Логика) (Сообщения) (Товары)  (Деньги)  (Выдача)   (Автоматика)
```

## Ключевые компоненты

1. **EventBus & EventJournal**
   - Все компоненты общаются друг с другом через события (`order_paid`, `delivery_success`, `message_send_failed`).
   - Если сервер падает, `EventJournal` (на базе SQLite WAL) восстановит недоставленные события при старте.

2. **OrderService (Orders)**
   - Единый источник истины по заказам. Плагины не меняют статусы заказов напрямую.
   - Контролирует переходы стейт-машины (NEW -> PAID -> PROCESSING -> DELIVERED -> COMPLETED).

3. **FinanceService (Ledger)**
   - Ведет финансовый учет (Income, Provider Cost, Profit, Commission).
   - Единственный компонент, имеющий право писать в таблицу Transactions.

4. **DeliveryService (Delivery)**
   - Отвечает за общение с поставщиками ключей/товаров. 
   - Вызывает `ChatService` (CCE) для отправки товара клиенту.

5. **Customer Communication Engine (CCE / ChatService)**
   - Интеллектуальный роутер сообщений клиентам. Защищает от спама и управляет сценариями (Автоответы, Уведомления об ошибках).

6. **Plugins Layer**
   - Песочница. Плагины слушают события и могут отправлять запросы в Фасад, но не имеют прямого доступа к базе данных или финансам.
